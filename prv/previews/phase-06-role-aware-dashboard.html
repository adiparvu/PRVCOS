Technical spec

Implementation architecture for the 4-variant dashboard system

🏗
Architecture
  • DashboardRouter server component — reads session.role, renders correct variant
  • 4 variant components: ExecutiveDashboard, ManagerDashboard, WorkerDashboard, SpecialistDashboard
  • Each variant is a server component — fetches its own data via scoped DB queries
  • Role-specific quick actions via resolveQuickActions(role)
  • All variants share glass card + section label primitives
🗄
Data layer
  • queryExecutiveKpis(companyId) — revenue, profit, projects, workforce, approval queue
  • queryManagerKpis(companyId, userId, role) — scoped to user's region/store/project
  • queryWorkerContext(userId) — shift, tasks, inbox count, team
  • querySpecialistContext(userId, role) — role-specific data sets
  • All queries behind cacheMemo — TTL 60s per company + role scope
⚡
Live data (Phase 06)
  • Dynamic Island receives live value from useLiveContext(role)
  • Supabase Realtime channels per company + role scope
  • Executive: revenue ticks, approval arrivals
  • Manager: target progress, staff clock-in/out
  • Worker: shift updates, new task assignments
  • Specialist: pipeline events, new tickets
🔐
Security
  • Server component reads session before rendering — no client-side role check
  • Each DB query scoped to companyId + userId + role
  • Scope validation in @prv/db query layer — not in component
  • Sensitive KPIs (revenue, payroll) only in executive + manager variants
  • All renders go through withGates in API routes
